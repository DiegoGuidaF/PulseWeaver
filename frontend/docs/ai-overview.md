# Frontend Architecture & Stack Overview

## System Context

- **Backend:** Go (Chi, SQLite) embedded binary.
- **Frontend Deployment:** SPA embedded into the Go binary (`dist` folder).
- **API Strategy:** Schema-first. We do not manually write API types; we generate them from `./api/api.yaml` via openapi
  generators.

## Core Tech Stack

| Component        | Technology               | Decision Reasoning                                   |
|:-----------------|:-------------------------|:-----------------------------------------------------|
| **Framework**    | React 18+                | Standard, robust ecosystem.                          |
| **Build Tool**   | Vite                     | Fast HMR, simple configuration.                      |
| **Language**     | TypeScript (Strict)      | Critical for low maintenance and refactoring safety. |
| **Styling**      | Tailwind CSS v4          | Utility-first, standard in modern React.             |
| **Components**   | shadcn/ui                | Copy-paste ownership, accessible, based on Radix UI. |

| **Client State** | URL Params / React State | No Redux/Zustand unless absolutely necessary.        |
| **Routing**      | React Router             | Standard SPA routing.                                |
| **Icons**        | Lucide React             | Consistent, clean icon set.                          |

## Data Flow Architecture

1. **Server State:** Managed exclusively by `TanStack Query`.
    - **Query & mutation options:** The frontend uses `@hey-api/openapi-ts` with the `@tanstack/react-query` plugin. Query options (e.g. `getDevicesOptions()`, `getDeviceAddressesOptions({ path })`) and mutation options (e.g. `createDeviceMutation()`, `addAddressMutation()`) are generated in `frontend/src/lib/api/@tanstack/react-query.gen`. Use these so HTTP, validation, and error shaping stay in the generated layer.
    - **Query keys:** Use generated keys from the same file: `getDevicesQueryKey()`, `getDeviceAddressesQueryKey({ path: { device_id } })`, `getCurrentUserQueryKey()`, etc. Custom keys in `@/lib/api-client/queryKeys.ts` are only for endpoints not yet covered by the generated React Query layer.
    - **Mutations:** Use `useMutation({ ...xxxMutation(), onSuccess, onError })`; feature code only adds UI behavior (toasts, invalidation, navigation). Call `mutate()` with the full options shape (e.g. `mutate({ body: values })`, `mutate({ path: { device_id, address_id } })`).
    - **Error Handling:** Always use Sonner toasts for mutation errors (`onError` with `toErrorMessage(err)` from `@/lib/api-client`).
2. **API Layer:**
    - SDK, types, and React Query glue: `@/lib/api` and `@/lib/api/@tanstack/react-query.gen` (generated by `@hey-api/openapi-ts` from `./api/api.yaml`). Do not modify files in `@/lib/api/`.
    - Config and helpers: `@/lib/api-client` (toApiError, toErrorMessage, client config). Query keys for OpenAPI-defined endpoints come from the generated file above.
    - Use generated SDK and query/mutation options; Zod schemas from `@/lib/api/zod.gen`.
3. **Component State:**
    - Local `useState` for UI interaction (modals, inputs).
    - URL search params for shareable state (filters, pagination).
